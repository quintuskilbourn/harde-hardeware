# Construction — the masked-EVM RTL

The hardware side of the [side-channel-proof EVM](../README.md): the masked RV64 **256-bit-native**
datapath, as Verilog, each primitive verified secure in the glitch+transition-robust probing model. The
verification framework these are checked against lives in [`../LeanSec`](../LeanSec); this directory is the
**constructions** — the actual masked circuits.

## Verified 256-bit masked primitives

| Primitive | Top module | Verdict doc |
|---|---|---|
| AND — nonlinear gadget-of-record (O-PINI2) | `MSKand_opini2_d2.v` | leaf (SILVER, in `../LeanSec`) |
| ADD / SUB | `top_add256_opini2.v` | `RESULT_ADD256.md` |
| ISZERO / EQ | `iszero256.v`, `eq256.v` | `RESULT_ISZERO256.md`, `RESULT_EQ256.md` |
| MUL (`A*B mod 2^256`) | `mul256.v` | `RESULT_MUL256.md` (independently audited) |
| DIV / MOD | `div256.v`, `divevm256.v` | `RESULT_DIV256.md`, `RESULT_DIVEVM.md` |
| Keccak-f[1600] (full permutation) | `keccakf1600.v` | `RESULT_KECCAKF1600.md` |
| Keccak-256 sponge | `sponge1600.v` | `RESULT_SPONGE1600.md` |
| masked storage (2-share scratchpad) | `store256x8.v` | `RESULT_STORE256.md` |
| constant-product swap (the anchor) | `swap256.v`, `dexloop256.v` | `RESULT_SWAP256.md` |

The `*16` / `keccakf100/200/400` modules are narrow dev versions used to bring up each design cheaply
before the full-width build.

**The swap** is assembled but its *whole-netlist* MATCHI run exceeds the exact tool at scale (the
verification-tractability wall) — it is discharged by **composition** (each primitive is verified once; the
[composition theorem](../LeanSec/Composition/Pipeline.lean) covers the wiring), which is the project's
answer to whole-core verification.

## How it's built
Every 256-bit value is carried as two shares. Linear structure (XOR/NOT and the adder dataflow) is
share-local; the one nonlinear primitive (AND) is the **O-PINI2** gadget `MSKand_opini2_d2.v`,
*instantiated* (never inlined) so the composition checker sees the boundary. Every primitive is built from
that gadget + the verified adder.

Each ships with negative controls (`*_rndreuse.v` = reused randomness, `*_recomb.v` = recombined shares)
that **must fail** verification, a `gen_*.py` generator, a `synth_*.ys` yosys script, a `tb_*.v` testbench,
a `run_*.sh` matrix runner, and a `RESULT_*.md` verdict.

## Reproduce
Needs [yosys](https://github.com/YosysHQ/yosys) + [MATCHI](https://github.com/cassiersg/matchi) on PATH.
```
./run_mul256.sh mul256            # -> exit 0  (secure)
./run_mul256.sh mul256_rndreuse   # -> exit 1  (randomness reuse caught)
./run_mul256.sh mul256_recomb     # -> exit 1  (share recombination caught)
```
A primitive is accepted only if the O-PINI top PASSes, **both** negative controls FAIL, and the checker
provably read the gadget instances (not a flattened netlist).

## Scope
First-order (d=1) glitch+transition-robust probing security. Synthesized netlists (`*_synth*`) and MATCHI
work dirs are regenerated by the scripts and are not committed. See [`../README.md`](../README.md) for the
whole-project context and roadmap (higher order, couplings, the proof-carrying compiler).
